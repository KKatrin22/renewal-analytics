# Проект: Анализ клиентских пролонгаций и эффективности менеджеров отдела сопровождения.

---

## Описание проекта

Проект посвящён анализу эффективности пролонгации клиентских договоров и оценке качества работы менеджеров отдела сопровождения.  

Цель проекта — определить, насколько эффективно сотрудники удерживают клиентов после завершения проектов, выявить слабые места в процессе пролонгации и подготовить управленческую отчётность для руководителя отдела сопровождения.

---

## Бизнес-задача

Руководителю отдела сопровождения клиентов требовалось:
- оценить эффективность менеджеров отдела сопровождения по продлению клиентских проектов;
- выявить сотрудников с лучшими и худшими показателями;
- определить месяцы с провалами в пролонгации;
- понять, на каком этапе теряется большая часть клиентов;
- сформировать основу для управленческих решений по улучшению удержания.

Для оценки использовались два ключевых коэффициента:

**Коэффициент пролонгации 1-го месяца**  
Показывает, какой объём клиентов был продлён сразу после завершения проекта.  
Формула:  
`Коэффициент = Отгрузка в первый месяц после завершения / Отгрузка последнего месяца проекта`

---

**Коэффициент пролонгации 2-го месяца**  
Показывает, какая часть клиентов была возвращена во второй месяц, если не была продлена в первый.  
Формула:  
`Коэффициент = Отгрузка во второй месяц / База проектов без пролонгации в первый месяц`

## Логика расчета

### Синхронизация данных и выделение отчетного периода

После того как оба информационных потока (основной файл `prolongations` и финансовый файл `financial_data`) были подготовлены, они объединяются методом `Left Outer Join` по ключевому полю `[id]`.

На стыке двух таблиц вычисляется ключевая метрика модели — `month_diff` (разница в месяцах между финансовым периодом `MonthKey_fin` и периодом закрытия `MonthKey_close`).

Далее применяется фильтрация целевого окна (`month_diff` от 0 до 2).

### Когортное разделение данных и формирование структуры таблиц

Для проведения глубокого анализа качества и скорости пролонгаций очищенный массив данных разделяется на три изолированные аналитические когорты (таблицы) по значению метрики `month_diff`:

#### Шаг 1. Определение базы завершённых проектов (`month_diff = 0`)

- Назначение: Данная таблица берется за основную опорную точку, в ней проводятся базовые расчеты и на нее накладываются метрики из таблиц задержек.

- Назначение: Используется как справочная когорта для расчета метрик дожима и эффективности менеджеров на первом этапе просрочки.

**Результат:** список проектов с пролонгацией первого месяца.

### Итог архитектуры

После разделения данных на три изолированные когорты (Base, M1, M2) в соответствии с временным лагом (`month_diff`), основной аналитический цикл продолжается внутри таблицы Base. Цель этого этапа — собрать финансовые метрики (отгрузки) по каждому проекту, сопоставить периоды их завершения с периодами фактических пролонгаций и подготовить агрегированные данные для расчета Коэффициентов 1 и 2 в разрезе менеджеров отдела сопровождения (AM).

#### Шаг 4. Агрегация проектных отгрузок по периодам пролонгации
Перед тем как объединить данные, внутри изолированных таблиц M1 и M2 выполняется предварительная подготовка и агрегация отгрузок на уровне проектов. Это необходимо для исключения дублирования строк при слиянии:

1. **Группировка и суммирование транзакций (1. Table.Group):** Внутри каждой таблицы (M1 и M2) данные схлопываются по связке `[id]` (ID проекта) и `[AM]` (менеджер отдела сопровождения). Все промежуточные финансовые поступления по конкретному проекту суммируются:
   - В таблице M1 формируется столбец общего дохода за первый месяц просрочки.
   - В таблице M2 формируется столбец общего дохода за второй месяц просрочки.

2. **Переименование полей (2. Table.RenameColumns):** Полученные агрегированные столбцы переименовываются в уникальные целевые метрики: `M1amount` и `M2amount` соответственно.

**Результат:** структуры таблиц M1 и M2 приведены к уникальному виду «один проект — одна строка суммарной отгрузки», запускается основной цикл консолидации в таблице Base.

- **Агрегация базового периода (Table.Group в таблице Base):** Данные таблицы Base группируются по связке `[id]` (ID проекта), `[AM]` (менеджер отдела сопровождения) и `[MonthKey_close]` (последний месяц реализации проекта). Все отгрузки последнего месяца реализации суммируются в столбец `[BaseAmount]`. На выходе получается строго одна уникальная строка на каждый завершившийся проект.

- **Интеграция пролонгаций 1-го месяца (Шаги 2–4 в Base):** Через левое соединение (`LeftOuter`) по уникальному полю `[id]` к базовому проекту подтягивается подготовленная метрика `[M1amount]` из таблицы M1. Если проект не был пролонгирован в первый месяц, системные пустоты `null` принудительно заменяются на `0`.

- **Интеграция пролонгаций 2-го месяца (Шаги 5–7 в Base):** Аналогичным образом через соединение `LeftOuter` по ключевому полю `[id]` подтягивается метрика `[M2amount]` из таблицы M2. Отсутствие отгрузок на второй месяц также переводится из `null` в `0`.

**Результат:** сформирована единая транзакционная матрица проектов. По каждому уникальному `id` проекта в одной строке видны: отгрузка в месяц завершения (`BaseAmount`), отгрузка при пролонгации в 1-й месяц (`M1amount`) и отгрузка при пролонгации во 2-й месяц (`M2amount`).

#### Шаг 5. Расчет компонентов Коэффициента 2 (Поздняя пролонгация)
Коэффициент 2 оценивает эффективность работы с проектами, которые не были пролонгированы в первый месяц, но получили отгрузку во второй.

Чтобы исключить из расчетов проекты, успешно продленные в первый месяц, и изолировать базу для оценки затяжной просрочки, в таблице Base создаются два расчетных поля с жесткой условной логикой:

- **Выделение целевой отгрузки («Числитель_Коэф2»):**  
  `if [M1amount] = 0 then [M2amount] else 0`

- **Определение потенциала базы («Знаменатель_Коэф2»):**  
  `if [M1amount] = 0 then [BaseAmount] else 0`

- **Формула KPI 2:**  
  `Коэффициент 2 = Σ(Числитель_Коэф2) / Σ(Знаменатель_Коэф2)`

Алгоритм проверяет первый месяц просрочки. Если отгрузка в первый месяц равна нулю (`M1amount = 0`), система берет сумму, которую менеджер смог дожать на второй месяц (`M2amount`), и отправляет её в числитель. Если проект уже был успешно продлен в первый месяц (`M1amount > 0`), он исключается из расчета (приравнивается к 0).

Базой (знаменателем) для таких проектов выступает сумма их отгрузки в последний месяц реализации (`BaseAmount`). Проекты, успешно продленные в первый месяц, сюда также не попадают.

Типизация: Созданные поля переводятся в формат `number` для корректного агрегирования.

В результате расчёта были сформированы три аналитические когорты:
- **Base** — база завершённых проектов;
- **M1** — пролонгации первого месяца;
- **M2** — пролонгации второго месяца.

#### Шаг 6. Финальная агрегация данных и формирование управленческой отчётности
После расчёта всех компонентов на уровне отдельных проектов была выполнена финальная консолидация данных.

**Цель этапа:** перевод расчётов с уровня отдельных ID проектов на уровень управленческой аналитики.

Что было сделано:

1. **Агрегация по менеджерам и месяцам**  
   Данные были сгруппированы по:
   - менеджерам отдела сопровождения
   - месяцу завершения проекта (`MonthKey_close`)

   Для каждой группы рассчитаны:
   - `Общий_BaseAmount` — общий объём завершённых проектов;
   - `Общий_M1Amount` — объём пролонгаций в первый месяц;
   - `Итого_Числитель_Коэф2` — сумма пролонгаций второго месяца;
   - `Итого_Знаменатель_Коэф2` — база для расчёта второго коэффициента.

**Результат:** сформирована единая агрегированная KPI-модель для дальнейшего анализа в Excel и Power BI.

---

2. **Расчёт KPI 1**  
   Формула:  
   `KPI1 = Общий_M1Amount / Общий_BaseAmount`  
   Показывает долю клиентов, продлённых сразу после завершения проекта.

---

3. **Расчёт KPI 2**  
   Формула:  
   `KPI2 = Итого_Числитель_Коэф2 / Итого_Знаменатель_Коэф2`  
   Показывает эффективность работы с клиентами, не продлёнными в первый месяц.

---

4. **Формирование временного представления**  
   Для удобства восприятия числовой период (`YYYYMM`) был преобразован в читаемый календарный формат.

   Пример:

   Это позволило сделать отчёт понятным для руководителя.

**Результат:**  
Сформирована итоговая управленческая таблица, позволяющая анализировать:
- эффективность каждого менеджера;
- динамику пролонгаций по месяцам;
- проблемные периоды;
- качество удержания клиентов.

**Итог этапа:**  
Сформирован единый аналитический датасет, который стал базой для дальнейшего построения управленческой отчётности в Excel (сводные таблицы) и интерактивной визуализации в Power BI.

## 10. Основные результаты

### Отчёт «Весь отдел»

При расчёте показателей пролонгации за 2023 год использовались проекты, завершившиеся с ноября 2022 по декабрь 2023 года. Это позволило корректно рассчитать коэффициенты пролонгации для первых месяцев 2023 года.

Анализ показал неравномерную динамику пролонгаций в течение года. Наиболее высокие показатели первого месяца наблюдались в январе (66%), сентябре (61%), ноябре (52%) и декабре (47%).  
В период с февраля по май показатели первого месяца были существенно ниже, что может указывать на сезонность либо снижение активности работы с клиентами.

Коэффициенты второго месяца значительно ниже и в большинстве случаев не превышают 15%, что подтверждает: основная часть клиентов принимает решение о продлении в первый месяц.  
Во втором полугодии наблюдается положительная динамика.

**Вывод:** первый месяц после завершения проекта является ключевым этапом удержания клиентов.

**Вывод:** закрепление клиентов за ответственными менеджерами и использование успешных практик повторных коммуникаций может повысить общий уровень удержания.

## 11. Бизнес-выводы и рекомендации

На основе проведённого анализа были сформулированы следующие рекомендации:

1. **Усилить работу с клиентами до завершения проекта**  
   Большая часть успешных пролонгаций происходит сразу после завершения проекта, поэтому критически важно начинать коммуникацию с клиентом заранее.

2. **Сфокусировать контроль на первом месяце пролонгации**  
   Первый месяц является ключевым окном удержания клиента. Именно в этот период вероятность продления максимальна.

3. **Внедрить отдельный процесс возврата клиентов второго месяца**  
   Для клиентов, не продливших проект в первый месяц, рекомендуется разработать отдельный сценарий повторного контакта и возврата.

4. **Масштабировать практики наиболее эффективных менеджеров отдела сопровождения**  
   Анализ показал значительный разрыв в результатах между сотрудниками. Практики менеджеров с высокими коэффициентами пролонгации могут быть использованы как стандарт работы отдела.

5. **Внедрить регулярный мониторинг KPI пролонгации**  
   Для своевременного выявления отклонений и оперативного управления удержанием клиентов рекомендуется использовать регулярную отчётность и дашборды.
